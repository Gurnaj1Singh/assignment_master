● DAY 1 — BLOCK 1A: The Models Layer

  File 1: models/base.py

  Let's start here. This 35-line file is the bedrock of your entire database.

  from sqlalchemy.orm import declarative_base
  Base = declarative_base()

  ---
  WHAT is declarative_base()?

  Base is a Python class factory. When you write:

  class User(TimestampMixin, SoftDeleteMixin, Base):
      __tablename__ = "users"
      id = Column(...)

  SQLAlchemy's metaclass machinery scans User at import time, reads every
  Column(...) attribute, and registers it in Base.metadata — a central registry
  of all table definitions.

  The analogy: Base is the architect's blueprint registry. Every model class
  that inherits from it is like handing a new floor plan to that registry. When
  you call Base.metadata.create_all(engine), SQLAlchemy takes all registered
  blueprints and issues CREATE TABLE statements in one shot.

  ---
  HOW it works under the hood

  Base = declarative_base()
       │
       └─► Returns a class whose metaclass is `DeclarativeMeta`
           DeclarativeMeta.__init__ runs for every subclass
           It inspects class attributes, finds Column() objects,
           builds a Table() object, registers it in Base.metadata

  Python's __init_subclass__ / metaclass hooks fire at class definition time,
  not at object instantiation. This is why you never call User() to create the
  table — the schema is registered the moment Python parses the class body.

  ---
  WHY TimestampMixin?

  Look at this:

  class TimestampMixin:
      created_at = Column(DateTime(timezone=True), server_default=func.now(),
  ...)
      updated_at = Column(DateTime(timezone=True), server_default=func.now(),
  onupdate=func.now(), ...)

  server_default=func.now() — This does NOT set the value in Python. It tells
  SQLAlchemy to emit DEFAULT NOW() in the CREATE TABLE SQL. The database engine
  sets the timestamp, not your Python process. This is critical for distributed
  systems where multiple app servers have different clocks.

  onupdate=func.now() — SQLAlchemy fires this client-side when an UPDATE is
  issued via the ORM. But note: if you update rows via raw SQL (UPDATE users SET
   ...), updated_at will NOT change. This is a known limitation — production
  systems often use a DB-level trigger instead.

  The Failure Test: "What if we removed TimestampMixin from all models?"

  You'd lose audit trails forever. No way to answer: "When was this submission
  made? Which submissions were created after the deadline? When was this
  classroom created?" Every compliance audit, every dispute resolution becomes
  impossible. Timestamps are non-negotiable in any system that touches academic
  integrity.

  ---
  WHY SoftDeleteMixin?

  class SoftDeleteMixin:
      is_deleted = Column(Boolean, default=False, nullable=False, index=True)

  Hard delete = DELETE FROM users WHERE id = ? — the row is gone forever.

  Soft delete = UPDATE users SET is_deleted = TRUE WHERE id = ? — the row exists
   but is hidden.

  Why it matters for your system specifically:
  - A student submits, professor reviews plagiarism report, then the submission
  is "deleted" — but you still need that submission's TextVector rows for
  historical similarity comparisons.
  - Legal: Universities may be required to retain academic records for 7 years.
  - Audit: "Show me all submissions that were deleted" — impossible with hard
  delete.

  Notice index=True on is_deleted. Every query filters by this column. Without
  the index, every query becomes a full table scan.

  The Failure Test: "What if we used hard deletes?" Every TextVector for a
  deleted submission cascades away. Future plagiarism checks miss those
  sentences. A student could delete their submission after being caught,
  eliminating the evidence.

  ---
  The Mixin Composition Pattern

  class User(TimestampMixin, SoftDeleteMixin, Base):

  Python's MRO (Method Resolution Order) resolves left-to-right. TimestampMixin
  columns come first, then SoftDeleteMixin, then Base's machinery runs. The
  mixins have no __init__ — they're pure column containers. SQLAlchemy's
  metaclass merges all columns from the MRO into the final Table object.

  Why not put timestamps and is_deleted directly into each model? DRY (Don't
  Repeat Yourself). You have 5 models. Without mixins, you'd write those same 3
  columns 5 times. One change (e.g., adding a deleted_by column to
  SoftDeleteMixin) propagates to all models automatically.

  ---
  UUID Primary Key — Why not Integer?

  id = Column(UUID(as_uuid=True), primary_key=True, default=uuid.uuid4)

  UUID(as_uuid=True) — PostgreSQL has a native uuid column type. as_uuid=True
  tells SQLAlchemy to map it to Python's uuid.UUID object, not a raw string.

  default=uuid.uuid4 — Note: this is uuid.uuid4 (a function reference), NOT
  uuid.uuid4() (a call). SQLAlchemy calls this function each time a new row is
  created. If you wrote default=uuid.uuid4(), every row would get the same UUID
  — generated once at class definition time.

  Why UUID over auto-increment integer?

  ┌─────────────────────────────────┬───────────────────────────────────────┐
  │           Integer PK            │                UUID PK                │
  ├─────────────────────────────────┼───────────────────────────────────────┤
  │ Exposes record count (/users/1  │ Opaque — no information leakage       │
  │ → "only 1 user")                │                                       │
  ├─────────────────────────────────┼───────────────────────────────────────┤
  │ Sequential — IDOR attacks easy  │ Random — cannot guess adjacent IDs    │
  │ (/users/2, /users/3)            │                                       │
  ├─────────────────────────────────┼───────────────────────────────────────┤
  │ Hard to shard across databases  │ Globally unique — merge two databases │
  │                                 │  trivially                            │
  ├─────────────────────────────────┼───────────────────────────────────────┤
  │ Fast index (narrow, sequential) │ Slightly wider index, but random      │
  │                                 │ insertion causes page splits          │
  └─────────────────────────────────┴───────────────────────────────────────┘

  For a system dealing with student records and academic integrity, IDOR 
  (Insecure Direct Object Reference) protection alone justifies UUIDs.

  The Failure Test: "What if we used integer PKs?" A student who submits
  assignment gets submission_id=42. They can immediately enumerate
  /submission-detail/41, /submission-detail/43 and read other students'
  plagiarism reports. That's a data breach.

  ---
  The role Column — A Hidden Gap

  role = Column(String(20), nullable=False)

  Right now, role is a free-text string. If someone inserts role="admin" or
  role="PROFESSOR" (uppercase) directly into the DB, your require_role(user, 
  "professor") check silently fails — they get 403 forbidden on everything.

  Production fix: use a PostgreSQL ENUM type:
  from sqlalchemy import Enum
  role = Column(Enum("student", "professor", name="user_role"), nullable=False)
  The DB itself rejects any value not in that set. This is called "pushing
  constraints down to the database" — defence in depth.

  ---
  Relationships — back_populates vs backref

  memberships = relationship("ClassroomMembership", back_populates="student",
  cascade="all, delete-orphan")

  back_populates requires you to define the relationship on both sides
  explicitly:
  - User.memberships → "I point to ClassroomMembership"
  - ClassroomMembership.student → "I point back to User"

  backref would auto-create the reverse side. It's a shortcut, but it's implicit
   — the reverse relationship exists but you can't see it in the code. With
  back_populates, both sides are visible, which is why it's preferred in
  production codebases.

  The analogy: back_populates is like a two-way contract both parties sign.
  backref is like giving someone a key to your house without telling them — it
  works, but nobody reading the code can see it.

  ---
  cascade="all, delete-orphan" — What does this actually do?

  When you delete a User via SQLAlchemy ORM:
  - cascade="all" → also deletes their ClassroomMembership rows and Submission
  rows
  - cascade="delete-orphan" → if you remove a Submission from user.submissions
  collection (without deleting the User), that orphaned Submission also gets
  deleted

  But wait — your models also have ondelete="CASCADE" on the ForeignKey:
  ForeignKey("users.id", ondelete="CASCADE")

  That's the database-level cascade. If someone runs DELETE FROM users WHERE 
  id=? in raw SQL, the DB handles the cascade. The ORM cascade handles it when
  you use db.delete(user).

  Having both is correct — defence in depth. ORM cascade for application-level
  deletes, DB cascade as a safety net for raw SQL operations.
